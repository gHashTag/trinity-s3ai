GOLDEN BRIDGE — Rust canvas UI (ring 4)

Build a structural bridge between data and geometry — without collapsing it. Pure-Rust render model + Canvas2D shell via wasm-bindgen. The game state lives in ring4_canvas::state::AppState; JavaScript only forwards mouse/key events and translates render primitives into draw calls.

Honesty notice. GOLDEN BRIDGE is a hypothesis-discovery puzzle, not a proven Theory of Everything. A high "bridge strength" only means the board passes more of the illustrative consistency checks than alternatives. Every tile is gated by a claim status; the worst such status is rendered next to the total. A falsified tile collapses the bridge regardless of the rest.
Fallback — non-interactive

Static GOLDEN BRIDGE layout snapshot

You are seeing the static snapshot because the WebAssembly bundle for the interactive UI is not loaded. The bundle is intentionally not checked into the repository (it is build output — shipping it would defeat the Rust-as-source-of-truth contract documented in docs/CANVAS.md). Fresh clones and sandboxed previews therefore land on this view. The deployed GitHub Pages build runs the live wasm shell.

Snapshot of the initial GOLDEN BRIDGE layout — empty board, default catalog, no benchmark mode. Piers, golden deck, recipe rail, towers, and score panel are drawn from the same Rust render primitives the wasm shell uses. Clicks do nothing in this mode.

How to enable the interactive build 
cd games/trinity_fold
wasm-pack build crates/ring4_canvas --target web --features wasm \
  --out-dir ../../web/canvas/pkg

# Regenerate the static snapshot:
cargo run -p ring4_canvas --example snapshot_svg -- \
  web/canvas/snapshot.svg

See docs/CANVAS.md for the rationale (Rust as source of truth, no JS mirror of game logic), including the GitHub Pages workflow that ships the live wasm canvas under games/trinity_fold/web/canvas.
